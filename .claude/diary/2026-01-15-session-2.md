## Design Decisions Made

### TDD-First Approach
- **Decision**: Write all tests before implementation
- **Why**: User explicitly requested TDD; tests document expected behavior and will catch regressions

### Test File Organization
- **Decision**: Organize tests by feature area (store, api, jobs, ingest)
- **Why**: Matches the package structure; easier to find tests for specific functionality

### Integration Tests vs Unit Tests
- **Decision**: Use integration tests with real database for store layer
- **Why**: Store operations involve SQL; mocking would miss actual query behavior

### testutil Package
- **Decision**: Create shared `internal/testutil` package with common helpers
- **Why**: Reduce duplication across test files (TestDB, CreateTestProject, pointer helpers)

### Test Database Connection Handling
- **Decision**: Skip tests gracefully if database unavailable
- **Why**: Tests should work in CI (with DB) but not crash locally if DB not running

## Code Patterns and Decisions

- **Go test pattern**: `func TestFeature_Scenario(t *testing.T)` with clear scenario comments
- **Helper pattern**: `strPtr()`, `intPtr()` etc. for creating pointers to literals
- **Database pattern**: `testDB(t)` returns pool or skips test
- **Python pattern**: `with patch()` for mocking, `pytest.raises()` for exceptions
- **Cleanup pattern**: `t.Cleanup()` for automatic teardown
