## Design Decisions Made

### Router Composition Pattern (Critical)
- **Decision**: Remove `/api/health` from OpenAPI spec, route it directly in Chi router
- **Why**: `HandlerWithOptions` applies middleware to ALL OpenAPI routes. Keeping health in OpenAPI would require path-based bypass logic in auth middleware, which is complex and error-prone. Router composition is cleaner.

### Env-Only Configuration for Phase 2
- **Decision**: Use environment variables only (DATABASE_URL, PORT, HOST), ignore `config.example.yaml`
- **Why**: The YAML config file includes advanced features (proxy, metrics, multiple providers) that are future scope. Env-only is simpler for Phase 2 bootstrap.

### parent_span_id as String (Not UUID)
- **Decision**: Change OpenAPI Span schema from `format: uuid` to plain `string` for parent_span_id
- **Why**: Database stores external span IDs as strings. UUID format would require complex lookup to convert external ID to internal UUID. Direct string mapping is simpler and sufficient for span tree visualization.

### Limit/Offset Pagination (Not Page)
- **Decision**: UI should use existing `limit`/`offset` API parameters, not add a `page` parameter
- **Why**: OpenAPI already defines `limit` (default 50) and `offset` parameters. Adding `page` would require API changes. Using existing params avoids drift.

### Multi-Tenant Scoping for ALL Endpoints
- **Decision**: All data endpoints (not just ListTraces/ListSessions/Ingest) must scope by project ID
- **Why**: GetTrace and ListSpansByTrace were initially missing from scoping tasks. Without project verification, any valid API key could fetch any trace by ID, breaking multi-tenancy isolation.

## Challenges Encountered

### Issue 1: OpenAPI Schema Mismatch
- **Problem**: API Span schema has `parent_span_id` with `format: uuid`, but DB stores external string IDs. Mapper was dropping the field entirely.
- **Resolution**: Explicitly documented schema change to remove UUID format, use plain string, direct mapping.

### Issue 2: Auth/Router Wiring Conflict
- **Problem**: Initial design assumed `ChiServerOptions.Middlewares` applies selectively. Actually, it applies to ALL OpenAPI routes including health.
- **Resolution**: Changed architecture to router composition - health outside OpenAPI, protected routes inside.

### Issue 3: Incomplete Multi-Tenant Scoping
- **Problem**: Initial tasks only listed 3 handlers for project scoping, missing GetTrace and ListSpansByTrace.
- **Resolution**: Added explicit scenarios for all 5 data endpoints in authentication spec.

### Issue 4: Pagination Parameter Drift
- **Problem**: Design doc example showed `?page=${page}` but API uses `limit/offset`.
- **Resolution**: Updated design doc and tasks to explicitly use `limit=${PAGE_SIZE}&offset=${offset}`.

## Solutions Applied

1. **Router Composition**: Created clear separation - health public, OpenAPI protected via group middleware
2. **Schema Extension Tasks**: Added Spec 0 as pre-requisite for OpenAPI changes before implementation
3. **Explicit Type Changes**: Documented exactly what to change in schema (remove `format: uuid`)
4. **Direct Mapping**: Specified mapper should do direct string copy, no UUID lookup

## Notes

### Key Insight: Discovery Before Planning
The Spec 0 (Repository Discovery) was critical. It revealed:
- All 6 API handlers already implemented (just not wired)
- Fx framework in go.mod but completely unused
- All rollup columns already exist in schema
- TanStack Query already installed in web UI
- Python SDK has generated types but no client

This discovery prevented "inventing" work that already exists and focused the plan on wiring, not building.

### Lesson: Schema vs Implementation Drift
The `parent_span_id` issue showed a common pattern: OpenAPI schema can drift from actual implementation. The schema said UUID but the DB stored strings. Always verify schema matches actual data flow.

### Multi-Agent Review Value
Having another agent review the proposal caught issues (pagination params, router wiring) that the initial planning missed. Iterative review is valuable for complex proposals.
