## Design Decisions Made

### PostgreSQL plainto_tsquery uses AND semantics
- **Decision**: Use AND semantics for multi-word searches
- **Why**: `plainto_tsquery('english', 'hello world')` produces `'hello' & 'world'` (AND, not OR)
- **Impact**: Search for "checkout flow" returns traces containing BOTH words, not either

### Index replacement scope narrowed to traces table only
- **Decision**: Only replace `idx_traces_started_at` and `idx_traces_server_received` with project_id-leading versions
- **Why**: Verified actual SQL queries:
  - `spans.sql`: All queries filter by `trace_id` only (ListSpansByTrace, etc.)
  - `events.sql`: All queries filter by `trace_id` only
  - `batches.sql`: No query filters by `project_id + status`
  - `traces.sql`: `ListTraces` DOES filter by `project_id` + order by start_time
- **Impact**: Avoids creating useless indexes that wouldn't help any existing queries

### Duration filter needs double COALESCE
- **Decision**: Use `COALESCE(end_time, now()) - COALESCE(start_time, server_received_at)`
- **Why**: Traces may have NULL start_time if client didn't send it, only server_received_at
- **Impact**: Duration filter won't silently drop traces missing client timestamps

### River job coalescing allows queued follow-up while running
- **Decision**: UniqueOpts excludes JobStateRunning (only dedupes Available/Retryable)
- **Why**: Allows one queued follow-up job while first is running, ensuring no lost updates
- **Impact**: Additional span ingests coalesce onto the queued job, not lost
