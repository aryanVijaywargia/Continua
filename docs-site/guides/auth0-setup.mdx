---
title: "Auth0 operator login"
description: "Gate the debugger UI behind Auth0 with an email allow-list."
---

By default the debugger expects a project API key entered on the Settings page. For
shared deployments with multiple operators, an internal tool, or anything you don't want
behind a single shared key, turn on Auth0.

## What Auth0 mode gives you

- The debugger UI redirects to Auth0 for sign-in.
- Only emails listed in `AUTH0_ALLOWED_EMAILS` can complete sign-in.
- The Settings page hides the local API key card and shows the operator account + a
  Sign out button.
- API key auth (`X-API-Key`) keeps working for SDK ingest. Operator auth gates the UI;
  ingest stays on its key.

## Required env vars

| Variable | Purpose |
| --- | --- |
| `AUTH0_ENABLED` | Master switch. Set to `true` to turn Auth0 on |
| `AUTH0_DOMAIN` | Auth0 tenant domain (e.g. `your-tenant.us.auth0.com`): host only, no scheme |
| `AUTH0_CLIENT_ID` | Auth0 SPA application client id |
| `AUTH0_AUDIENCE` | API audience identifier from your Auth0 API config |
| `AUTH0_ALLOWED_EMAILS` | Comma-separated allow-list. At least one required when enabled |

All four are required when `AUTH0_ENABLED=true`: the server fails to start otherwise,
listing the missing variables.

## End-to-end setup

### 1. Create the Auth0 application

In your Auth0 tenant:

1. **Applications → Create Application → Single Page Application.**
2. Set **Allowed Callback URLs** and **Allowed Logout URLs** to your debugger origin
   (e.g. `https://debugger.example.com`).
3. Set **Allowed Web Origins** to the same value.
4. Note the **Domain** and **Client ID**: you'll need them as env vars.

### 2. Create the Auth0 API

1. **APIs → Create API.**
2. Name it, pick a unique identifier (e.g. `https://continua/api`): this is your
   `AUTH0_AUDIENCE`.
3. Leave the signing algorithm at the default.

### 3. Configure the Continua server

```bash
export AUTH0_ENABLED=true
export AUTH0_DOMAIN=your-tenant.us.auth0.com
export AUTH0_CLIENT_ID=abc123...
export AUTH0_AUDIENCE=https://continua/api
export AUTH0_ALLOWED_EMAILS=alice@example.com,bob@example.com
```

Restart the server. Health check still works at `GET /api/health` without
authentication.

### 4. Sign in

Open the debugger. You should be redirected to Auth0, sign in, and land back at the
debugger. The Settings page shows your email + a Sign out button.

If your email is not on `AUTH0_ALLOWED_EMAILS`, the server returns 403 after Auth0
sign-in completes.

## Adding or removing operators

Auth0 enforces sign-in; Continua enforces the allow-list. Update the allow-list and
restart the server to add or remove operators:

```bash
export AUTH0_ALLOWED_EMAILS=alice@example.com,bob@example.com,carol@example.com
```

There's no admin UI for managing the allow-list today. Pair Auth0 with your own access
review process.

## Combining Auth0 with public demo mode

You can't run both at once. If `PUBLIC_DEMO_ENABLED=true`, demo mode wins and the UI
serves a single read-only project to anyone. Use that for landing / evaluation pages and
keep Auth0 for operator deployments.

## Troubleshooting

- **`AUTH0_DOMAIN must be a valid domain`**: drop `https://` from the value. Domain is
  host-only.
- **`AUTH0_ALLOWED_EMAILS must include at least one email address`**: at least one
  syntactically-valid email is required when Auth0 is on.
- **Redirect loop after sign-in**: check that the **Allowed Callback URLs** in Auth0
  exactly matches the debugger origin you're hitting (no trailing slash mismatch, no
  protocol mismatch).
- **403 after Auth0 sign-in completes**: your email isn't on `AUTH0_ALLOWED_EMAILS`.
